What is Procure-to-Pay?

End-to-end process from identifying a procurement need to paying the vendor

P2P (Procure-to-Pay) is the complete business process cycle in SAP MM that starts when a department identifies a need for goods or services and ends when the vendor is paid. It spans three SAP modules: MM (Materials Management), FI-AP (Accounts Payable), and optionally CO (Controlling) for cost assignment.

Core Process Steps

  • 1. Purchase Requisition (PR)
  • 2. Source Determination
  • 3. Request for Quotation (RFQ)
  • 4. Purchase Order (PO)
  • 5. Goods Receipt (GR)
  • 6. Invoice Verification (MIRO)
  • 7. Payment (F110)

Key Integration Points

  • MM ↔ FI: Automatic accounting documents on GR and MIRO
  • MM ↔ CO: Cost center / order assignment on PO
  • MM ↔ WM: Transfer orders for warehouse-managed storage locations
  • MM ↔ SD: Stock transfers, returns
  • MM ↔ PP: MRP-driven purchase requisitions

Organisational Structure Required

  • Client (mandant)
  • Company code → FI entity
  • Plant → production/storage site
  • Purchasing organisation → legal purchasing entity
  • Purchasing group → buyer group
  • Storage location → physical store

Master Data Prerequisites

  • Vendor master (XK01) — all views
  • Material master (MM01) — purchasing, MRP, accounting views
  • Purchasing info record (ME11)
  • Source list (ME01) — if mandatory
  • Account determination / valuation class

Movement Types (Inventory Management)

Movement types control how goods movements are posted and which accounts are updated. Configured via OMJJ.

Key principle: Every goods movement in MIGO/MB1A/MB1B is controlled by a 3-digit movement type. The movement type determines: stock type affected, account keys for FI posting, print output, and whether a reversal is allowed.

Release (Approval) Strategy

SAP uses a classification-based release strategy for both PR and PO approval workflows.

The release strategy in SAP MM uses the Classification system (not workflow). Characteristics (e.g., total value, plant, purchasing group) are assigned to a class, which is linked to a release strategy defining who must approve at which step. Path: SPRO → MM → Purchasing → Purchase Requisition → Release Procedure.

Release Strategy Components

  • Release group — groups of strategies (e.g. for PR, for PO)
  • Release codes — individual approver codes (e.g. 01=Buyer, 02=Manager)
  • Release indicators — status per step (blocked, released, final release)
  • Release prerequisite — defines the approval sequence
  • Classification class — links characteristics to strategy determination
  • Characteristics — CEBAN fields: GSWRT (total value), WERKS (plant), EKGRP (purch. group)

Transaction Codes

  • ME54N — Release individual PR
  • ME55 — Collective release (PRs)
  • ME28 — Release PO
  • ME29N — Release PO (new)
  • CL24N — Display classification assignments
  • CT04 — Maintain characteristics
  • CL02 — Maintain classes
